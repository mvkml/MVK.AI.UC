### Decision
1. **Layering/DI shape approved**: Controller constructor-injects `UserBL`; `UserBL` constructor-injects `UserRepository`; `UserRepository` constructor-injects `AiHrDbContext`. Matches existing constructor-injection-ready classes already written.
2. **No interfaces yet**: Register concrete `UserBL`, `UserRepository`, `UserValidationService` directly. `AI.HR.BL`/`DAL`/`Repoistories` are otherwise empty placeholder projects with no established interface convention, and there's a single consumer per class today — introducing `IUserBL`/`IUserRepository`/`IUserValidationService` now is premature abstraction. Revisit per-feature once tests need mocking or a second implementation appears.
3. **Lifetime: Scoped for all three** (`UserRepository`, `UserBL`, `UserValidationService`). `AiHrDbContext` is Scoped (default `AddDbContext`); `UserRepository` holds it directly so must also be Scoped to avoid a captive-dependency bug. `UserBL` follows transitively. `UserValidationService` is stateless (Transient/Singleton would also work) but kept Scoped for registration-pattern consistency.
4. **UserValidationService relocated**: Move from `AI.HR.Api/Services/` to `AI.HR.BL/UserValidationService.cs` (namespace `AI.HR.BL`). Validation of business rules is business logic, not an API-layer concern, and per ADR002 `AI.HR.Api` should stay a thin HTTP layer. Requires adding project reference `AI.HR.Api -> AI.HR.BL` (not yet present).
